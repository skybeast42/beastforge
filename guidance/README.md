# guidance/

**Instructions CC must follow** for doing the work well — distinct from
`background/` (passive context, may consult, not loaded) and from `CLAUDE.md`
(template machinery, generic). Loaded *contextually*: CC reads the relevant
doc before doing that kind of work.

## The three base files (always present)

- **`01_standing.md`** — universal, stable rules. Ships filled. Edit rarely.
- **`02_derived.md`** — this project's specifics (tooling, style, testing).
  Ships as a skeleton marked `Status: uninitialised`; CC proposes its content
  from `MASTERPLAN.md` and the User approves. Then it is a normal owned file.
- **`03_user.md`** — how to work with *this* User (preferences, working style).
  Ships empty; CC fills it as it learns the User. Always-on, like `01_standing`.

## Domain docs (added on top, not shipped)

Specialist methodology — e.g. a model-evaluation or analysis-methodology doc.
Project-specific and User/research-maintained; a domain folder may be authored by
hand, ported in from another project, or **generated by `/ground`** (which distils
dropped material into `guidance/<topic>/`). **No naming convention** — the numeric
prefixes keep the base files sorted on top; domain docs sort below. No metadata: git
is the history.

## Precedence

`01_standing` (universal floor) → `02_derived` (project specifics) → domain
doc (specialist). Compose where they agree; the more specific doc wins where
they conflict.
