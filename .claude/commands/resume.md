description: Entry workflow — orient (or cold-start ingest dumped code) and propose the next action
---

Resume work on this project.

1. Read `CLAUDE.md` (the rules), `STATE.md`, `MASTERPLAN.md`, and the **always-on guidance baseline** (`guidance/01_standing.md` + `guidance/03_user.md`); **run `/inbox`** (the asks relay — lead with any open **urgent** asks; `/inbox` also surfaces *your* `operator` inbox to read & reply through this beast); note the template version (`.template_version`). If `tracks/legacy/README.md` exists, read it (cheap) — it manifests prior-version work parked at migration; **be aware of it, consult on demand, don't re-shape it**.
2. **Scan `guidance/` for domain topics.** A domain topic is any `guidance/` subfolder that **contains a `distilled.md`** (the marker of a grounding-produced topic — the base files `01_standing.md`/`02_derived.md`/`03_user.md` are files, not folders, so they never match; a half-built folder without a `distilled.md` doesn't register). Topics are **detected, not declared**. For each such `guidance/<topic>/`, read its `README.md` (one paragraph — cheap). Note the topic's name + its "**Read when:**" activation line in the summary; **do not** auto-load each `distilled.md` (consumed on demand when the work matches). **Discovery is authoritative — there is no registry to reconcile** (a topic is active the moment it has a `distilled.md`, however it arrived); just report what's present. No "register?" prompt.
3. Run the **task-integrity check** and the **reconciliation checks** (CLAUDE.md → "The task-integrity check" + "Reconciliation checks"). Report any brief-less tasks, name near-misses, or reconciliation warnings (unlogged work / reproducible-execution gaps / task gap / command-table parity / TODOLIST done-pileup). Do **not** auto-correct or delete; if a name near-miss is found, stop and ask the User.
4. Detect entry mode — check in this order, **precedence matters** (so virgin never shadows a content dump). Key on `STATE.md`'s **Status** flag, which `/setup` flips `uninitialised`→`active`:
   - `STATE.md` **`Status: active`** (or otherwise populated) → summarise where things stand (active tasks / apps / tracks, open tickets, open `TODOLIST.md` items, open asks, open audits in `tracks/audits/`), then map `MASTERPLAN.md`'s conceptual steps to propose the next action. **Special case — set up but no strategy yet:** if `MASTERPLAN.md` is still a skeleton (a just-`/setup`'d or just-`/spawn`ed beast — identity, nameplate, color, lineage all done, strategy not), the beast is **past `/setup`** → propose **`/masterplan`** (or diving into the first task). **Never re-propose `/setup`** — it has already run; re-running it would only re-ask the two questions.
   - else, content present in `work/`, `apps/`, or `tracks/` → cold-start ingest: scan, report, and *propose* (do not force) bootstrapping `STATE.md`/`MASTERPLAN.md` and wrapping loose reproducible code into stub-brief tasks (`Origin: imported`). `apps/` stays framework-driven.
   - else **virgin** (`STATE.md` **`Status: uninitialised`**, only placeholders) → fresh template: whatever the User typed (even "hello"), point them to `/setup` (and `/ground` if they already have domain material) and offer to walk through it. A suggestion, never interrogation.
5. Do **not** auto-load `background/` — consult it only if needed for the proposed next action. The two always-on guidance docs (`01_standing`, `03_user`) are read at step 1; other guidance docs (`02_derived`, domain-specific) are read only when the proposed action is code or domain work. Domain topics discovered at step 2 are read on demand when their *Read when:* triggers match.

Keep the summary tight. End with a single concrete proposed next action for the User to confirm.
