Bootstrap this project from the template. Keep it minimal ‚Äî **the questions below plus name/description**; do not turn this into a wizard. **All entry paths converge here** (a plain `git clone`, GitHub **"Use this template"**, the `getstarted.md` chat on-ramp, and `/spawn`), so this is the one place `origin` and the habitat home get settled ‚Äî coherently, with no special-casing.

1. **Initialize the README.** If `README.md` has a `<!-- BEAST-README-BELOW -->` marker near the top ‚Äî a fresh clone of the **public** BeastForge repo carries a SkyBeast landing section + that marker *above* the generic body ‚Äî strip everything from the top of the file **through the marker line**, and delete the launch-only files if present (`getstarted.md`, `banner.png`, `banner.svg`). **First note the template's own URL** from that landing section / `getstarted.md` ‚Äî step 3 needs it. 2. **Habitat home (detect-and-offer; optional).** Look for a `.habitat/` home (locate per `/habitat` step 1 ‚Äî ancestor-walk by the `habitat.md` marker, name-agnostic). **If absent, offer once** to create it via `/habitat init` (uninitialized ‚Äî `origins_root` left blank); declined ‚Üí the beast runs fine standalone. `.habitat/` is **optional**: its only jobs are the **template location** (for `/spawn`) and **`origins_root`**. *Note the name ‚Äî `origins_root`, not "origins": it is the **root directory under which** bare origins are created (blank = no bare repos at all).* A lone or test beast needs neither job ‚Äî so the first beast often skips this, and the offer naturally resurfaces when a second beast appears.

3. **Settle the `origin` (git-exception ‚Äî needs a shell with `git`).** Like `/spawn` and `/tokens`, this single step shells out (`git remote`, `git init`); if no shell/git is available, skip it and tell the User the commands to run. Read `git remote get-url origin` (if any) and apply **one rule** ‚Äî the template's identity is the URL noted in step 1 (or `template_source` from `.habitat`):
   - **Compare by normalized identity, not raw string.** Canonicalize *both* `origin` and the template URL to `host/owner/repo` before comparing ‚Äî strip the scheme, normalize `git@host:owner/repo` ‚áî `https://host/owner/repo`, and drop any trailing `.git` or `/`. (So `https://github.com/skybeast42/beastforge`, `‚Ä¶/beastforge.git`, and `git@github.com:skybeast42/beastforge.git` all match.) The same-owner case is decided by **repo name**: `skybeast42/garden` ‚âÝ `skybeast42/beastforge` ‚Üí *not* the template. If the template identity is only a **local path** (not a URL), resolve *its* own `origin` (`git -C <template_source> remote get-url origin`) and compare that, or treat the step-1 URL as authoritative; **if neither side resolves to a comparable URL, default to the safe branch (keep ‚Äî never auto-strip on uncertainty).**
   - **origin == the template** (a plain clone of BeastForge ‚Äî the working copy also carries BeastForge's entire dev history, which isn't this beast's story) ‚Üí **offer a clean slate**: `rm -rf .git && git init -b main` so the beast starts its *own* history with no template remote. **Set the branch to `main` explicitly** (`-b main`, or `git branch -m main` on older git) ‚Äî a bare `git init` inherits the host default (often `master`), but the published repo and the beast pool standardize on `main` (as `/spawn` does). Recommend yes (it's the whole point of cloning) ‚Äî and say plainly what it does: *discards BeastForge's dev history (intended for a clone ‚Äî your beast starts at commit one; re-clone to undo)*. It is the one destructive action here, so confirm once. (It is gated on the template match, so it can never fire on a kept own-repo origin.)
   - **origin is something else** (your own repo ‚Äî e.g. created by GitHub's "Use this template") ‚Üí **keep it.** This beast is already yours and pushable; do nothing.
   - **no origin** (e.g. `/spawn` already wired one, or a deliberately local beast) ‚Üí do nothing.

   After a clean slate (or for any origin-less beast), give it a home **only if one is configured**: if `.habitat` has `origins_root` set ‚Üí create the local bare (`git init --bare -b main <origins_root>/<name>.git`), wire `origin`, **make the initial commit** (`git add -A && git commit` ‚Äî the clean-slate path has no commits yet, and a push needs one), push, and say so. If `origins_root` is **blank/absent** ‚Üí leave the beast **local-only** and tell the User plainly that the origin is theirs to choose: set `origins_root` in `.habitat` and re-run, add their own `git remote add origin <url>`, or ‚Äî cleanest for a hosted backup from day one ‚Äî start instead from GitHub's **"Use this template"** (which hands you your own remote). **Never invent a remote.**

   **End-state commit (one rule, both paths).** A **clean-slated** beast needs its initial commit (above) ‚Äî that becomes commit one of its own history. A beast with a **kept origin** (your own "Use this template" repo) already has history; leave `/setup`'s edits **unstaged for the User to review and commit** (the standing *propose-a-commit, User-approved* rule) ‚Äî don't auto-commit over their repo. Either way, never leave a clean-slated beast with zero commits *and* a configured remote.
